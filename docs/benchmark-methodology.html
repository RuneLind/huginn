1. Production Flow: Chrome Extension to Analysis

In production, Jira issue analysis flows through four systems. The benchmark recreates the middle part (knowledge search + analysis) in isolation so we can measure and iterate.

2. Benchmark Pipeline

The benchmark has two modes. In search-quality mode (default), it scores what the knowledge base returns. In analysis mode (--analyze MODEL), it also generates an analysis and scores the model's synthesis ability.

3. Query Extraction

The benchmark simulates what an AI agent would search for by extracting queries from the Jira issue content. Each issue generates up to 3 queries, searched against each collection.

4. Scoring Rubric

Each evaluation is scored on 5 dimensions by an LLM judge (Claude Sonnet). Dimensions have different weights reflecting their importance for producing a useful work plan.

Domain Understanding
2.0x
Do results explain lovvalg, BUC/SED, forordninger?
Technical Context
2.0x
Are architecture docs, APIs, services found?
Related Work
1.5x
Are epics, linked issues, cross-refs found?
Actionability
1.5x
Could an agent create a concrete work plan?
Noise Ratio
1.0x
Proportion of results that are relevant

How the judge works

The judge receives a structured prompt with the issue content, all search results (with content snippets), and any graph context. It returns a JSON object with 5 scores and a reasoning string.

# Judge response format
{"domain_understanding": 4, "technical_context": 2,
 "related_work": 5, "actionability": 3, "noise_ratio": 4,
 "reasoning": "Strong domain coverage with relevant docs..."}

The judge is invoked via claude --model claude-sonnet-4-6 -p <prompt> with a 120-second timeout. Failed calls receive neutral scores of 3.0.

5. Autoresearch Iteration Loop

Inspired by Karpathy's autoresearch: fixed inputs, variable treatments, measurable outputs. Each iteration changes one variable, runs the benchmark, and keeps or discards the change based on score delta.

Key insight from autoresearch The fixed time-budget concept maps perfectly: each benchmark run takes the same ~20 minutes regardless of configuration, making results directly comparable. The metric (weighted judge score) is configuration-independent, just like val_bpb in the original.

6. The Content Visibility Fix

A critical bug in the benchmark caused the wiki to appear useless. The fix changed scores dramatically.

sequenceDiagram participant API as Knowledge API participant BM as Benchmark Script participant Judge as LLM Judge rect rgb(254, 202, 202) Note over API,Judge: BEFORE FIX - bug API->>BM: Response includes matchedChunks with content, breadcrumb, graph BM->>BM: Looks for top-level matchedContent key - WRONG KEY BM->>BM: Gets empty string BM->>Judge: Title only, no content snippet Judge->>Judge: Sees blank wiki results Judge-->>BM: Wiki scores LOW - looks like noise end rect rgb(187, 247, 208) Note over API,Judge: AFTER FIX API->>BM: Same response BM->>BM: Reads matchedChunks content - CORRECT KEY BM->>BM: Also extracts breadcrumb, heading, graph context BM->>Judge: Rich result with content, section, graph entities Judge->>Judge: Sees relevant domain knowledge Judge-->>BM: Wiki scores HIGH - real value end

Impact on scores (3-issue pilot)

ConfigBefore FixAfter FixChange
baseline2.693.23+20%
with-wiki2.523.60+43%
full-knowledge2.603.46+33%
The wiki went from -6.3% (hurting!) to +11.6% (best config) The entire "wiki adds noise" finding was a measurement artifact. The curated domain wiki is actually the most valuable collection for analysis — the judge just couldn't see its content.

What the fix extracts from each search result

# Before: only top-level fields (mostly empty)
title = r.get("title", "untitled")
snippet = r.get("matchedContent", "")[:200]  # always empty!

# After: nested content + metadata + graph
title = r.get("title", "untitled")
breadcrumb = r.get("breadcrumb", "")
chunks = r.get("matchedChunks", [])
snippet = chunks[0].get("content", "")[:300]
heading = chunks[0].get("heading", "")
graph_ctx = r.get("graph_context", [])

8. Usage Reference

Running the benchmark

# Full benchmark (14 issues x 3 configs, ~20 min)
.venv/bin/python scripts/evaluation/jira_analysis_benchmark.py

# Specific issues only
.venv/bin/python scripts/evaluation/jira_analysis_benchmark.py \
    --issues MELOSYS-7219 MELOSYS-6432

# Specific configs
.venv/bin/python scripts/evaluation/jira_analysis_benchmark.py \
    --configs baseline with-wiki

# Analysis mode: generate analysis with a model, then score it
.venv/bin/python scripts/evaluation/jira_analysis_benchmark.py \
    --analyze claude-sonnet-4-6 \
    --configs full-knowledge

# Dry run (search only, no LLM judge)
.venv/bin/python scripts/evaluation/jira_analysis_benchmark.py --dry-run

# Custom judge model and output
.venv/bin/python scripts/evaluation/jira_analysis_benchmark.py \
    --judge-model claude-sonnet-4-6 \
    --output scripts/evaluation/results/experiment-42.json
