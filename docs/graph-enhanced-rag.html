Huginn: Graph-Enhanced RAG

How knowledge graphs, community detection, and LLM entity extraction improve search beyond standard vector similarity

The Problem with Standard RAG

Standard Retrieval-Augmented Generation treats documents as isolated chunks of text. A vector search for "trygdeavgift" finds chunks that contain that word or similar embeddings — but it has no understanding that trygdeavgift is calculated from Næringsinntekt, paid to NAV, regulated by Folketrygdloven, and processed in Melosys.

Huginn solves this by layering three graph-based enhancements on top of hybrid search:

  1. LLM-extracted knowledge graphs — entities and relationships discovered by running a local LLM over every document
  2. Community detection — automatic clustering of semantically related documents using the Louvain algorithm
  3. Graph-enhanced search pipeline — entity detection, query expansion, and result enrichment at search time

How Entity Detection Works

When a query arrives, Huginn detects entities using two complementary methods:

1. Regex patterns (domain-specific)

Hardcoded patterns for known entity formats:

  • LA_BUC_01 → BUC entity
  • A001, X007 → SED entities
  • artikkel 13 nr. 1 → Article entity
  • MELOSYS-6587 → Jira issue

Fast, predictable, zero false positives for known formats.

2. LLM graph label matching

Case-insensitive substring matching against all entity:* node labels:

  • "trygdeavgift" → entity:trygdeavgift
  • "Claude Code" → entity:claude_code
  • "regelmotor" → entity:regelmotor
  • "Gosys" → entity:gosys

Catches implicit entities that regex can't anticipate.

Query Expansion via Graph Traversal

Once entities are detected, Huginn follows 1-hop edges in the knowledge graph to collect related terms. These terms are appended to the original query, dramatically improving recall.

The expanded query finds documents that discuss trygdeavgift indirectly — for example, a Confluence page about "Næringsinntekt i utlandet" that never mentions the word "trygdeavgift" but is highly relevant.

Graph Context Enrichment

After search results are ranked, each result is annotated with graph_context — structured relationship information from the knowledge graph. This gives the LLM (or human reader) immediate understanding of how concepts connect.

Search result: "Kodeverk i Melosys.md"
graph_context:
• MELOSYS (Organization) | part_of Jira, uses SED, part_of EØS, part_of Årsavregning
• Kodeverk (Concept) | uses WSDL, uses ISO2, backend uses | 2 mentions
Search result: "Claude Code just Built me an AI Agent Team"
graph_context:
• Claude Code (Product) | built_by Anthropic, uses MCP, uses Bash, related_to GitHub | 190 mentions
• MCP (Technology) | related_to Salesforce, connects_to Claude Code | 34 mentions

This context is exposed through the search API and consumed by MCP-connected AI agents, giving them structured domain knowledge alongside the raw text chunks.

Community Detection: Automatic Document Clustering

Independent from the knowledge graph, Huginn runs Louvain community detection on the document similarity matrix to automatically discover thematic clusters within each collection.

LLM Entity Extraction Pipeline

The knowledge graph is built offline by sending each document to a local LLM (Ollama) for structured entity and relationship extraction.

Key design decisions

  • Fully local — Uses Ollama with qwen3.5, no data leaves the machine
  • Incremental — Per-document cache (.cache.json) allows stopping and resuming
  • Cross-document merging — Entities appearing in multiple documents get aggregated mention counts, revealing importance
  • Structured output — Uses Ollama's format: "json" and think: false for reliable parsing
  • Private repo storage — NAV data goes to huginn-nav, personal data to huginn-jarvis

Running extraction

# Extract entities for a collection (can stop and resume)
uv run scripts/knowledge_graph/extract_entities_llm.py --collection melosys-confluence-v3

# Test with a small sample first
uv run scripts/knowledge_graph/extract_entities_llm.py --collection jira-issues --limit 20

# Use the larger model for better quality
uv run scripts/knowledge_graph/extract_entities_llm.py --collection jira-issues --model qwen3.5:35b

Inspiration: Microsoft GraphRAG

This architecture is inspired by Microsoft's GraphRAG research, which showed that standard vector-search RAG fails on two key tasks:

  1. Connecting disparate information — when answering requires traversing relationships across documents
  2. Holistic sensemaking — when the user asks broad questions about themes across a corpus

Huginn adapts these ideas to a fully local architecture:

GraphRAG (Microsoft)Huginn
OpenAI API for entity extractionLocal Ollama (qwen3.5)
Leiden algorithm for communitiesLouvain algorithm (networkx)
